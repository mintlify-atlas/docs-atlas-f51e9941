## Overview

This example implements a simple yet efficient order matching engine for a trading system. The engine matches buy and sell orders using **price-time priority**, demonstrating fundamental concepts used in real-world trading platforms.

### What You'll Learn

- Order matching algorithms and priority rules
- Using `BTreeSet` for efficient order book management
- Partial order fills and trade execution
- Price-time priority implementation
- Testing trading logic with comprehensive test cases

## Engine Architecture

The matching engine uses a **dual BTreeSet** structure to maintain separate buy (bid) and sell (ask) order books:

```rust
struct MatchingEngine {
    bid: BTreeSet<Order>,
    ask: BTreeSet<Order>,
}
```

### Why BTreeSet?

The `BTreeSet` data structure was chosen for its balance of simplicity and performance:

<CardGroup cols={2}>
  <Card title="O(log n) Operations" icon="gauge-high">
    Both insertion and removal operations are O(log n), providing consistent performance as the order book grows.
  </Card>
  
  <Card title="Automatic Sorting" icon="arrow-down-1-9">
    Orders are automatically maintained in sorted order, making it easy to find the best bid/ask prices.
  </Card>
  
  <Card title="Simple Implementation" icon="code">
    Rust's standard library provides a robust BTreeSet implementation with no external dependencies.
  </Card>
  
  <Card title="Memory Efficient" icon="memory">
    Balanced tree structure provides good memory locality compared to linked lists.
  </Card>
</CardGroup>

### Performance Characteristics

**Per Order Submission:**
- Insert order into BTreeSet: **O(log n)**
- Pop order from BTreeSet: **O(log n)**
- Matching process: **O(m log n)** where m is number of matches

<Accordion title="Alternative Implementations">
  **Vector-Based (Nasdaq-style):**
  - Uses a vector for all possible price levels
  - O(1) access to any price level
  - More complex to implement
  - Better performance for high-frequency trading
  
  **Doubly Linked List:**
  - Maintains pointers to current best price
  - Extremely performant for insertions at current price
  - More complex implementation
  - Better for market makers posting at best bid/ask
  
  This BTreeSet approach prioritizes **simplicity** and is suitable for most trading scenarios that don't require microsecond latency.
</Accordion>

## Order Structure

Orders are represented with price, quantity, side, and a unique identifier:

```rust
type Price = Decimal;
type Quantity = u64;
type OrderIdentifier = u64;

static NEXT_ORDER_ID: AtomicU64 = AtomicU64::new(1);

#[derive(Debug, PartialEq, Eq)]
struct Order {
    id: OrderIdentifier,
    price: Price,
    quantity: Quantity,
    side: Side,
}

#[derive(Debug, PartialEq, Eq)]
enum Side {
    Buy,
    Sell,
}
```

### Order Creation

```rust
fn new(price: Price, quantity: Quantity, side: Side) -> Self {
    assert!(price > Decimal::ZERO, "Price must be positive");
    Self {
        id: NEXT_ORDER_ID.fetch_add(1, std::sync::atomic::Ordering::SeqCst),
        price,
        quantity,
        side,
    }
}
```

<Note>
Order IDs are assigned atomically using `AtomicU64`, incrementing from 1. This ensures unique IDs even in multi-threaded environments and provides a natural time priority ordering.
</Note>

### Order Comparison

Orders implement custom comparison logic to support price-time priority:

```rust
impl PartialOrd for Order {
    fn partial_cmp(&self, other: &Self) -> Option<std::cmp::Ordering> {
        if self.price == other.price {
            // Earlier id should be matched first (time priority)
            self.id.partial_cmp(&other.id)
        } else {
            // Sort by price
            self.price.partial_cmp(&other.price)
        }
    }
}
```

**Priority Rules:**

<Steps>
  <Step title="Price Priority">
    Orders are first sorted by price. Best bids (highest price) and best asks (lowest price) have priority.
  </Step>
  
  <Step title="Time Priority">
    When prices are equal, orders with lower IDs (submitted earlier) are matched first.
  </Step>
</Steps>

## Order Matching Process

When a new order is submitted, the engine attempts to match it with existing orders on the opposite side:

```rust
pub fn submit_order(&mut self, order: Order) -> Vec<Trade> {
    match order.side {
        Side::Buy => self.bid.insert(order),
        Side::Sell => self.ask.insert(order),
    };

    self.match_orders()
}
```

### Matching Algorithm

<CodeGroup>
```rust Main Logic
fn match_orders(&mut self) -> Vec<Trade> {
    let mut trades = Vec::new();

    // Continue while both sides have orders
    while !self.bid.is_empty() && !self.ask.is_empty() {
        let bid_order = self.bid.last().unwrap();  // Highest bid
        let ask_order = self.ask.first().unwrap(); // Lowest ask

        if bid_order.price < ask_order.price {
            // No crossing orders
            break;
        }

        // Match found - pop both orders
        let mut bid_order = self.bid.pop_last().unwrap();
        let mut ask_order = self.ask.pop_first().unwrap();

        let qty_matched = std::cmp::min(bid_order.quantity, ask_order.quantity);

        // Determine trade price and party IDs
        let (trade_price, party_order_id, counterparty_order_id) =
            if bid_order.id > ask_order.id {
                (ask_order.price, bid_order.id, ask_order.id)
            } else {
                (bid_order.price, ask_order.id, bid_order.id)
            };

        // Update quantities
        bid_order.quantity -= qty_matched;
        ask_order.quantity -= qty_matched;

        // Reinsert partially filled orders
        if bid_order.quantity > 0 {
            self.bid.insert(bid_order);
        } else if ask_order.quantity > 0 {
            self.ask.insert(ask_order);
        }

        trades.push(Trade {
            price: trade_price,
            quantity: qty_matched,
            party_order_id,
            counterparty_order_id,
        });
    }

    trades
}
```

```rust Trade Structure
#[derive(Debug)]
struct Trade {
    price: Price,
    quantity: Quantity,
    party_order_id: OrderIdentifier,
    counterparty_order_id: OrderIdentifier,
}
```
</CodeGroup>

### Trade Price Determination

The trade price follows the **liquidity taker pays maker's price** rule:

<Tabs>
  <Tab title="Bid Came Later">
    If the buy order has a higher ID (came in later), it's the taker and trades at the ask price:
    
    ```rust
    if bid_order.id > ask_order.id {
        (ask_order.price, bid_order.id, ask_order.id)
    }
    ```
  </Tab>
  
  <Tab title="Ask Came Later">
    If the sell order has a higher ID (came in later), it's the taker and trades at the bid price:
    
    ```rust
    else {
        (bid_order.price, ask_order.id, bid_order.id)
    }
    ```
  </Tab>
</Tabs>

<Tip>
This pricing mechanism rewards liquidity providers (makers) who post resting orders by giving them a better price than the taker who crossed the spread.
</Tip>

## Design Decisions

### BTreeSet Choice

The BTreeSet approach was selected to balance multiple concerns:

<CardGroup cols={2}>
  <Card title="Simplicity" icon="star">
    Standard library implementation with no external dependencies or complex data structure management.
  </Card>
  
  <Card title="Performance" icon="rocket">
    O(log n) operations are acceptable for most trading scenarios and scale well to thousands of orders.
  </Card>
  
  <Card title="Correctness" icon="check">
    Well-tested standard library reduces bugs compared to custom implementations.
  </Card>
  
  <Card title="Maintainability" icon="wrench">
    Easy for other developers to understand and modify without specialized knowledge.
  </Card>
</CardGroup>

### Price-Time Priority

The engine implements standard **FIFO (First In, First Out)** matching:

1. **Price Priority**: Best prices always match first
   - Highest bids match before lower bids
   - Lowest asks match before higher asks

2. **Time Priority**: At the same price, earlier orders match first
   - Order ID serves as timestamp proxy
   - Simple and deterministic

<Warning>
In production systems, you'd use actual timestamps rather than sequential IDs to handle distributed order submission and clock synchronization.
</Warning>

### Atomic Order IDs

```rust
static NEXT_ORDER_ID: AtomicU64 = AtomicU64::new(1);
```

Using `AtomicU64` ensures thread-safe ID generation, making the engine ready for multi-threaded extensions:

- **Thread-safe**: Multiple threads can submit orders concurrently
- **Sequential**: IDs naturally represent submission order
- **Efficient**: Lock-free atomic operations
