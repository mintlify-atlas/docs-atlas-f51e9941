## Overview

This example demonstrates building a full-text search system using ParadeDB, a PostgreSQL extension that provides high-performance search capabilities. The implementation shows practical database integration patterns and CLI design in Rust.

### What You'll Learn

- Full-text search with ParadeDB's BM25 algorithm
- PostgreSQL integration using the `postgres` crate
- Database schema design for search
- CLI application structure with `clap`
- Article data modeling and fetching

## What is ParadeDB?

ParadeDB is a PostgreSQL extension that adds advanced full-text search capabilities using the BM25 ranking algorithm. It provides:

- **Fast full-text search** with relevance scoring
- **PostgreSQL integration** - works with existing Postgres installations
- **BM25 ranking** - industry-standard search algorithm
- **Native SQL syntax** - query using familiar SQL operators

<Note>
This example uses basic ParadeDB functionality. In production, you'd add more sophisticated ranking, highlighting, faceting, and other search features.
</Note>

### Database Connection

The `DbConnection` struct manages the PostgreSQL connection:

```rust
use postgres::{Client, NoTls};

pub struct DbConnection {
    client: Client,
}

impl DbConnection {
    pub fn new() -> Result<Self, postgres::Error> {
        let client = Client::connect(
            "host=localhost port=5433 user=postgres password=postgres123 dbname=parade",
            NoTls,
        )?;

        Ok(Self { client })
    }
}
```

## Schema Design

The database schema is designed for full-text search:

```rust
pub fn init_database(&mut self) -> Result<(), postgres::Error> {
    self.client.batch_execute(
        "
        CREATE TABLE IF NOT EXISTS articles (
            id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
            title TEXT NOT NULL,
            content TEXT NOT NULL,
            published_at TIMESTAMP WITH TIME ZONE NOT NULL
        );

        CREATE INDEX IF NOT EXISTS articles_search_idx 
        ON articles 
        USING bm25(id, title, content) 
        WITH (key_field='id');
        ",
    )?;
    Ok(())
}
```

<Tabs>
  <Tab title="Table Structure">

    | Column | Type | Description |
    |--------|------|-------------|
    | `id` | UUID | Unique identifier (auto-generated) |
    | `title` | TEXT | Article title (searchable) |
    | `content` | TEXT | Article body (searchable) |
    | `published_at` | TIMESTAMPTZ | Publication timestamp |
  </Tab>
  
  <Tab title="BM25 Index">
    **articles_search_idx:**
    
    ```sql
    CREATE INDEX articles_search_idx 
    ON articles 
    USING bm25(id, title, content) 
    WITH (key_field='id');
    ```
    
    - **Index type**: `bm25` (ParadeDB's search index)
    - **Indexed fields**: `title` and `content`
    - **Key field**: `id` (used for scoring)
    - **Algorithm**: BM25 ranking for relevance scoring
  </Tab>

<Tip>
The BM25 index enables fast full-text search with relevance scoring. ParadeDB automatically maintains this index as you insert, update, or delete articles.
</Tip>

## Article Data Model

Articles are represented with a clean, database-independent structure:

```rust
use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};
use uuid::Uuid;

#[derive(Debug, Serialize, Deserialize)]
pub struct Article {
    id: Uuid,
    title: String,
    content: String,
    published_at: DateTime<Utc>,
}

impl Article {
    pub fn new(id: Uuid, title: String, content: String, published_at: DateTime<Utc>) -> Self {
        Self { id, title, content, published_at }
    }

    pub fn id(&self) -> Uuid { self.id }
    pub fn title(&self) -> &str { &self.title }
    pub fn content(&self) -> &str { &self.content }
    pub fn published_at(&self) -> DateTime<Utc> { self.published_at }
}
```

<Note>
The Article struct is intentionally decoupled from the database, making it easy to:
- Serialize/deserialize from JSON
- Use with different storage backends
- Test without database dependencies
- Share between different parts of the application
</Note>

### Loading Articles from JSON

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct Articles {
    articles: Vec<Article>,
}

impl IntoIterator for Articles {
    type Item = Article;
    type IntoIter = std::vec::IntoIter<Article>;

    fn into_iter(self) -> Self::IntoIter {
        self.articles.into_iter()
    }
}

pub fn read_articles_from_disk(path: &str) -> Result<Articles, std::io::Error> {
    let file = File::open(path)?;
    let reader = BufReader::new(file);
    let articles: Articles = serde_json::from_reader(reader)?;
    Ok(articles)
}
```

## Inserting Articles

Articles are inserted using parameterized queries:

```rust
pub fn insert_article(&mut self, article: &Article) -> Result<(), postgres::Error> {
    self.client.execute(
        "INSERT INTO articles (id, title, content, published_at) VALUES ($1, $2, $3, $4)",
        &[
            &article.id(),
            &article.title(),
            &article.content(),
            &article.published_at(),
        ],
    )?;
    Ok(())
}
```

The update command reads articles from JSON and inserts them:

```rust
fn update_from_json(conn: &mut db::DbConnection) -> Result<(), SearchError> {
    let articles = fetcher::read_articles_from_disk("data/sample_articles.json")?;

    for article in articles {

    }

    Ok(())
}
```

## Full-Text Search

ParadeDB's search uses the `@@` operator with BM25 scoring:

```rust
pub fn search_articles(&mut self, query: &str) -> Result<Vec<Article>, postgres::Error> {
    let rows = self.client.query(
        "SELECT id, title, content, published_at, paradedb.score(id) 
         FROM articles 
         WHERE title @@ $1 OR content @@ $1 
         ORDER BY score DESC",
        &[&query],
    )?;

    Ok(rows
        .into_iter()
        .map(|row| {
            let id = row.get::<_, Uuid>("id");
            let title = row.get::<_, String>("title");
            let content = row.get::<_, String>("content");
            let published_at = row.get::<_, DateTime<Utc>>("published_at");

            Article::new(id, title, content, published_at)
        })
        .collect())
}
```

### Search Query Breakdown

<CodeGroup>
```sql Query Structure
SELECT 
    id, 
    title, 
    content, 
    published_at, 
    paradedb.score(id)  -- BM25 relevance score
FROM articles 
WHERE 
    title @@ $1         -- Match in title
    OR 
    content @@ $1       -- Match in content
ORDER BY score DESC;    -- Best matches first
```

```rust Scoring
// The @@ operator performs full-text search
// ParadeDB's BM25 algorithm scores each match:
// - Term frequency in document
// - Inverse document frequency across corpus
// - Document length normalization

paradedb.score(id)  // Returns BM25 score for ordering
```
</CodeGroup>

<Accordion title="How BM25 Scoring Works">
  BM25 (Best Match 25) is a probabilistic search algorithm that ranks documents based on:
  
  1. **Term Frequency (TF)**: How often query terms appear in the document
  2. **Inverse Document Frequency (IDF)**: How rare/common terms are across all documents
  3. **Document Length**: Normalizes scores for longer vs shorter documents
  
  This produces more relevant results than simple keyword matching, prioritizing:
  - Documents where query terms appear frequently
  - Rare/specific terms over common words
  - Shorter documents with good matches over long documents with scattered mentions
</Accordion>

## Error Handling

The application uses custom error types with `thiserror`:

```rust

pub enum SearchError {
```

```rust
fn main() -> Result<(), Box<dyn std::error::Error>> {
```

### Database Independence

The Article struct is decoupled from database implementation:

```rust
// Article doesn't know about PostgreSQL
pub struct Article {
    id: Uuid,
    title: String,
    content: String,
    published_at: DateTime<Utc>,
}

// Only DbConnection knows about database details
impl DbConnection {
    pub fn insert_article(&mut self, article: &Article) -> Result<(), postgres::Error> {
        // Database-specific code lives here
    }
}
```

This design makes it easy to:
- Switch database backends
- Test with mock data
- Serialize to different formats
- Use in different contexts
