## Overview

Rust has excellent built-in testing support. This guide covers unit testing, async testing, and test organization patterns extracted from real examples.

## Basic Unit Tests

Unit tests live in the same file as the code they test, inside a `#[cfg(test)]` module:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_submit_order_no_match() {
        let mut engine = MatchingEngine::new();
        let order = Order::new("10.0050".parse().unwrap(), 100, Side::Buy);
        let trades = engine.submit_order(order);
        assert_eq!(trades.len(), 0);
    }
}
```

<Note>
The `#[cfg(test)]` attribute ensures test code is only compiled when running `cargo test`, keeping your release binaries lean.
</Note>

<Tip>
Always reset global state in tests to prevent test interdependence. Tests should be able to run in any order.
</Tip>

## Testing Panics

Use `#[should_panic]` to test that code panics as expected:

```rust
#[test]
#[should_panic(expected = "Price must be positive")]
fn test_invalid_price() {
    Order::new(Decimal::ZERO, 100, Side::Buy);
}
```

This test from order_book/src/main.rs:365-368 verifies that creating an order with zero price panics with the correct message.

<Warning>
Only use `should_panic` for testing invariant violations. For normal error conditions, use `Result` and test with `assert!(result.is_err())`.
</Warning>

## Complex Test Scenarios

### Multi-Step Tests

From order_book/src/main.rs:250-282, testing sequential operations:

```rust
#[test]
fn test_submit_order_with_match() {
    reset_order_id();

    let mut engine = MatchingEngine::new();
    let bid_order = Order::new("10.0050".parse().unwrap(), 100, Side::Buy);
    let ask_order = Order::new("09.0030".parse().unwrap(), 50, Side::Sell);

    // Submit bid
    let trades = engine.submit_order(bid_order);
    assert_eq!(trades.len(), 0);

    // Submit ask. This should match the bid.
    let trades = engine.submit_order(ask_order);
    assert_eq!(trades.len(), 1);
    assert_eq!(trades[0].price, "10.0050".parse().unwrap());
    assert_eq!(trades[0].quantity, 50);
    assert_eq!(trades[0].party_order_id, 2);
    assert_eq!(trades[0].counterparty_order_id, 1);

    // Submit another ask order. Should match again.
    let ask_order = Order::new("09.0030".parse().unwrap(), 50, Side::Sell);
    let trades = engine.submit_order(ask_order);
    assert_eq!(trades.len(), 1);
    
    // Submit another ask order. Shouldn't match anymore.
    let ask_order = Order::new("09.0030".parse().unwrap(), 50, Side::Sell);
    let trades = engine.submit_order(ask_order);
    assert_eq!(trades.len(), 0);
}
```

This test verifies:
1. Orders don't match when prices don't cross
2. Matching produces correct trades
3. Partial fills work correctly
4. Order book state is maintained

    io::{AsyncReadExt, AsyncWriteExt},

### Async Test Patterns

From server.rs:80-115, creating an interactive test client:

```rust
async fn interactive_client(
    addr: &str,
    mut consumer: Receiver<String>,
    producer: Sender<String>,
) -> tokio::io::Result<()> {
    let mut stream = TcpStream::connect(addr).await?;
    let (mut reader, mut writer) = stream.split();
    let mut buffer = Vec::new();

    loop {
        buffer.fill(0);
        select! {
            result = reader.read_buf(&mut buffer) => {
                if result.is_err() {
                    break;
                }
                let msg_from_server = String::from_utf8_lossy(&buffer).to_string();
                producer.send(msg_from_server).await.unwrap();
            }
            msg = consumer.recv() => {
                writer.write_all(msg.unwrap().as_bytes()).await.unwrap();
            }
        }
    }

    Ok(())
}
```

## Test Isolation

### Separate Runtimes for Server Tests

From server.rs:122-128:

```rust
// Create a thread that runs its own tokio runtime
let _server_thread = std::thread::spawn(|| {
    // Create a new tokio runtime for this thread
    let runtime = Runtime::new().unwrap();
    
    // Run the async server on this runtime
    runtime.block_on(async { Server::start_server(addr).await });
});
```

This pattern prevents deadlocks by running the server in a separate runtime from the test clients.

## Property-Based Testing

Test edge cases and invariants:

```rust
#[test]
fn test_price_time_priority() {
    reset_order_id();

    let mut engine = MatchingEngine::new();

    // Create two asks at same price, different times
    let ask1 = Order::new("10.00".parse().unwrap(), 50, Side::Sell);
    let ask2 = Order::new("10.00".parse().unwrap(), 50, Side::Sell);
    engine.submit_order(ask1);
    engine.submit_order(ask2);

    // Submit matching bid
    let bid = Order::new("10.00".parse().unwrap(), 50, Side::Buy);
    let trades = engine.submit_order(bid);

    // Earlier order should match first (FIFO)
    assert_eq!(trades.len(), 1);
    assert_eq!(trades[0].counterparty_order_id, 1);  // First ask
}
```

## Best Practices

1. **Test Naming**: Use descriptive names that explain what's being tested
   - `test_submit_order_no_match` ✓
   - `test1` ✗

2. **Arrange-Act-Assert**: Structure tests clearly
   ```rust
   // Arrange: Set up test data
   reset_order_id();
   let mut engine = MatchingEngine::new();
   
   // Act: Perform the operation
   let trades = engine.submit_order(order);
   
   // Assert: Verify the result
   assert_eq!(trades.len(), 0);
   ```

3. **Test One Thing**: Each test should verify a single behavior

4. **Avoid Test Interdependence**: Tests should pass regardless of execution order

5. **Use Helper Functions**: Extract common setup into helper functions like `reset_order_id()`

## Common Pitfalls

<Warning>
**Time-Based Tests**: The chat test uses `std::thread::sleep(Duration::from_secs(1))` to wait for server startup. This is fragile - prefer synchronization mechanisms when possible.
</Warning>

<Warning>
**Port Conflicts**: When testing network code, either use random ports (`bind("127.0.0.1:0")`) or ensure tests don't run in parallel.
</Warning>

<Warning>
**Floating Point Comparisons**: Use decimal types for precise comparisons in financial calculations, not `f64`.
</Warning>
